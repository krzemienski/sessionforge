  <action>
Using the production URL from Plan 02 (referred to as PROD_URL below), run these verification commands:

**Step 1: Healthcheck (DEPL-06)**
```bash
curl -sf https://PROD_URL/api/healthcheck | python3 -m json.tool
```
Expected response (200 OK):
```json
{
  "status": "ok",
  "db": true,
  "redis": false,
  "timestamp": "..."
}
```
- `status` must be `"ok"` (not `"degraded"`)
- `db` must be `true` (Neon connected via pooled URL)
- `redis: false` is acceptable if Upstash Redis is not configured (app runs without it per D-07)
- HTTP status must be 200 (not 503)

**Step 2: Deployment Validation (DEPL-07)**
```bash
curl -sf https://PROD_URL/api/deployment/validate | python3 -m json.tool
```
Expected response (200 OK):
```json
{
  "status": "ok",
  "mode": "neon-managed",
  "checks": {
    "required": {
      "DATABASE_URL": { "present": true },
      "BETTER_AUTH_SECRET": { "present": true },
      "NEXT_PUBLIC_APP_URL": { "present": true }
    },
    "optional": { ... },
    "db": { "ok": true },
    "redis": { "configured": false, "ok": false }
  },
  "timestamp": "..."
}
```
- `status` must be `"ok"`
- `mode` must be `"neon-managed"` (confirms Vercel + Neon detection per DEPL-02)
- All `required` vars must show `present: true` (per DEPL-05)
- `db.ok` must be `true` (confirms runtime DB connectivity)
- `redis.configured: false` is acceptable if Upstash not configured

**Step 3: Smoke test the app (DEPL-01)**
```bash
curl -sf -o /dev/null -w '%{http_code}' https://PROD_URL/
```
Expected: HTTP 200 or 302 (redirect to login). NOT 500/502/504.

If DEPLOYMENT_VALIDATE_TOKEN was set in env vars, add the Authorization header:
```bash
curl -sf -H "Authorization: Bearer TOKEN" https://PROD_URL/api/deployment/validate | python3 -m json.tool
```

Record all three responses for the summary.
